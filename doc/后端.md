# 后端开发规划文档

> 项目：居民用电分析与节能建议系统
>  
> 适用范围：`backend/` Go 主服务
>  
> 当前目标：先完成后端主流程、数据存取、规则分析与占位集成，Python Robyn 后端（分类/预测/智能体）的具体实现后续接入

---

## 1. 文档目标

本文件用于固定后端开发的执行口径，避免后续开发过程中出现以下问题：

- 后端职责与 Python Robyn 后端边界不清
- 导入、分析、分类、预测、建议、报告的主流程顺序反复变动
- 前端联调字段已经冻结，但后端内部实现路径没有统一
- Python Robyn 后端尚未实现，导致后端主链路迟迟无法推进

本文件的作用不是替代需求文档和 API 文档，而是回答三个问题：

1. Go 主服务具体负责什么
2. 每条业务链路在后端内部如何流转
3. Python 后端尚未实现时，后端如何先把骨架搭起来

技术栈执行口径：

- Web 框架：`Gin`
- 数据访问：`GORM`
- 结构化日志：`Zap`
- 异步任务：第一版使用进程内任务执行器，`Asynq` 仅作为后续扩展方案

任务跟踪约定：

- `- [x]` 表示已完成并已落地
- `- [ ]` 表示尚未完成或仅停留在设计阶段
- 每完成一个后端里程碑或接口，应同步更新本文件中的勾选状态

## 2. 后端定位

Go 主服务是系统的主协调层，负责：

- 对前端提供 `/api/v1` REST API
- 管理数据集、分析结果、分类结果、预测结果、建议、聊天会话、报告记录
- 管理文件落盘路径与数据库元数据
- 执行规则型统计分析、峰谷分析、规则建议生成
- 在需要分类、预测、回测、智能问答时，调用外部服务
- 在外部服务失败时执行降级或返回可解释错误

Go 主服务不负责：

- 深度学习模型训练
- 深度学习推理算法本身
- 大模型推理与提示词执行本身
- 前端图表计算逻辑

边界结论：

- **统计分析、峰谷分析、规则建议** 属于 Go 主服务
- **TCN 分类、LSTM/Transformer 预测、回测、智能问答** 属于 Python Robyn 后端
- 其中 `classification`、`forecast`、`agent` 在 Python 侧按模块拆分，但统一通过一个 Robyn 服务对外提供接口

## 3. 文档优先级与执行口径

当前后端实现应优先遵循以下文档：

1. `doc/chinese_project.md`
2. `doc/api_design.md`
3. `doc/database_design.md`
4. `doc/model_data_pipeline.md`

若出现不一致，按以下规则处理：

- 需求范围以 `chinese_project.md` 为准
- 接口字段与响应结构以 `api_design.md` 为准
- 表结构与数据职责以 `database_design.md` 和 `schema.sql` 为准
- 建模输入输出规格以 `model_data_pipeline.md` 为准

### 3.1 当前已发现的不一致

`model_history_window_config` 的默认示例值在文档间存在冲突：

- `api_design.md` / `schema.sql` 示例值是 `classification_days = 7`、`forecast_history_days = 7`
- `model_data_pipeline.md` 冻结规则是：分类按 `1天` 样本，预测按 `3天 -> 1天`
- `api_design.md` 中模型服务 `/model/info` 也写明：分类最小长度 `96`，预测最小长度 `288`

后端当前执行口径固定为：

- `classification_days = 1`
- `forecast_history_days = 3`

理由：

- 这与当前冻结建模方案一致
- 这与模型输入长度 `96 / 288` 一致
- `7 / 7` 只能视为过期示例值，后续应同步修正文档和初始化脚本

## 4. 后端模块划分

建议采用标准分层结构，避免把所有逻辑堆到 handler 中。

推荐目录：

```text
backend/
├── cmd/api/
├── internal/
│   ├── config/
│   ├── bootstrap/
│   ├── http/
│   │   ├── handler/
│   │   ├── middleware/
│   │   └── router/
│   ├── service/
│   ├── repository/
│   ├── domain/
│   ├── analysis/
│   ├── advice/
│   ├── report/
│   ├── integration/
│   │   ├── modelclient/
│   │   └── agentclient/
│   ├── storage/
│   └── job/
└── pkg/
    ├── response/
    ├── errors/
    └── utils/
```

各层职责：

- `handler`：参数解析、响应封装、状态码映射
- `service`：业务编排、事务边界、调用 repository 和 integration
- `repository`：数据库读写
- `analysis`：纯规则统计、峰谷平分析、质量摘要计算
- `advice`：规则建议生成与降级建议生成
- `integration/modelclient`：调用 Python Robyn 后端中的模型相关路由
- `integration/agentclient`：调用 Python Robyn 后端中的智能体相关路由
- `storage`：上传文件、导出文件、JSON 详情文件的落盘与读取
- `job`：异步导入、重算、导出任务

补充约束：

- `repository` 默认基于 `GORM` 实现
- 复杂统计查询允许使用 `Raw SQL`
- 日志统一接入 `Zap`
- Gin 请求日志、中间件日志、GORM SQL 日志应尽量归并到同一套日志体系

## 5. 配置与初始化

后端启动时至少完成以下初始化：

1. 读取 `.env`
2. 初始化 MySQL 连接
3. 初始化文件存储根目录
4. 初始化系统配置读取器
5. 初始化模型客户端与智能体客户端
   - 两者可配置为同一个 Robyn 服务地址
6. 注册路由与统一中间件

建议环境变量：

- `APP_PORT`
- `APP_HOST`
- `MYSQL_DSN`
- `MODEL_SERVICE_BASE_URL`
- `AGENT_SERVICE_BASE_URL`
- `REQUEST_TIMEOUT_SECONDS`
- `DATA_UPLOAD_DIR`
- `OUTPUT_ROOT_DIR`

说明：

- `DATA_UPLOAD_DIR`、`OUTPUT_ROOT_DIR` 可与 `system_config` 中的目录配置联合使用
- 运行时优先级建议为：环境变量 > 数据库配置默认值
- 第一版允许 `MODEL_SERVICE_BASE_URL` 与 `AGENT_SERVICE_BASE_URL` 指向同一个 Robyn 服务，只是路由前缀不同

## 6. 数据与文件职责

后端必须严格执行“元数据入库，大对象落盘”。

数据库中保存：

- 数据集元信息
- 分析摘要
- 分类摘要
- 预测摘要（写入 `forecast_results.summary`）
- 建议摘要
- 会话与消息摘要
- 报告记录

文件系统中保存：

- 原始上传文件
- 清洗后的 `15min` 数据文件
- 数据质量报告 JSON
- 分析详情 JSON
- 预测明细 JSON
- 建议正文 JSON
- 长消息内容
- 导出报告文件

删除策略：

1. 先删除文件
2. 再删除数据库记录
3. 文件删除失败时记录错误并阻止数据库级硬删除，避免脏元数据

## 7. 核心业务流程

## 7.1 数据集导入流程

目标：实现 `POST /api/v1/datasets/import`

建议流程：

1. 接收 `multipart/form-data`
2. 校验文件类型，仅允许 `csv` / `xlsx`
3. 创建 `datasets` 记录，状态设为 `uploaded`
4. 将原始文件保存到 `uploads/datasets/`
5. 更新状态为 `processing`
6. 解析列映射；自动识别失败时返回 `COLUMN_MAPPING_REQUIRED`
7. 读取原始数据并完成字段标准化：
   - 时间解析
   - 单位转换到统一制式
   - 识别总用电量列与任意数量的电器子表列
   - 重采样到 `15min`
   - 缺失值处理
   - 异常值裁剪或标记
   - 基于已识别电器列计算 `active_appliance_count` 与 `burst_event_count`
8. 生成质量摘要与质量报告 JSON
9. 输出清洗后的标准化数据文件
10. 回填 `datasets`：
    - `processed_file_path`
    - `row_count`
    - `time_start`
    - `time_end`
    - `feature_cols`
    - `column_mapping`
    - `quality_report_path`
11. 触发统计分析计算
12. 状态更新为 `ready`

失败处理：

- 任一步失败，将 `datasets.status` 更新为 `error`
- `error_message` 写入可展示信息
- 已生成的中间文件允许保留，便于排查

实现建议：

- 对前端返回 `202 Accepted`
- 后台使用 goroutine 或任务执行器完成剩余导入流程
- 第一版不必引入复杂队列系统
- 导入协议不应写死 `Appliance1~Appliance9`；该限制仅属于 REFIT 训练数据，不属于主服务用户导入协议

## 7.2 统计分析流程

目标：实现 `GET /api/v1/datasets/{id}/analysis` 与 `POST /analysis/recompute`

规则分析由 Go 本地完成，不依赖模型服务。

计算内容：

- 总用电量
- 日均用电量
- 最大/最小负荷及时间
- 峰谷平 `kwh`
- 峰谷平占比
- 日趋势
- 周趋势
- 典型日曲线
- 峰谷饼图

流程：

1. 读取 `processed_file_path`
2. 读取 `peak_valley_config`
3. 调用 `analysis` 模块计算摘要与图表数据
4. 将摘要写入 `analysis_results`
5. 将图表明细写入 `outputs/analysis/*.json`
6. 返回可直接渲染的结构

说明：

- `analysis_results` 只保留当前结果，重算时执行覆盖更新
- 若数据集还未进入 `ready`，返回 `DATASET_NOT_READY`

## 7.3 分类推理流程

目标：实现 `POST /api/v1/datasets/{id}/classifications/predict`

流程：

1. 校验数据集状态为 `ready`
2. 读取清洗后的 `15min` 数据
3. 根据请求窗口抽取分类输入序列
4. 组装模型服务请求体
5. 调用模型客户端 `PredictClassification`
6. 将模型返回的 `prob_*` 字段收口为 `probabilities`，并生成 `explanation`
7. 将结果写入 `classification_results`
8. 返回最新分类结果

当前策略：

- 后端只负责抽取序列、调用外部服务、落库
- 模型服务返回原始分类结果；公共接口需要的 `probabilities` 和 `explanation` 由 Go 主服务整理
- 不在 Go 内实现 TCN 推理

占位策略：

- 当模型服务未接入时，客户端返回 `MODEL_SERVICE_UNAVAILABLE`
- 为了不中断联调，可增加一个本地 stub 模式，返回固定结构的假数据

## 7.4 时序预测流程

目标：实现 `POST /api/v1/datasets/{id}/forecasts/predict`

流程：

1. 校验数据集状态
2. 根据 `forecast_start` / `forecast_end` 反推所需历史窗口
3. 从处理后数据中抽取最近 `3天` 历史序列
4. 组装模型服务 `forecast` 请求
5. 调用模型客户端
6. 根据模型返回的原始 `predictions[96]` 生成带时间戳的预测序列与 `ForecastSummary`
7. 将摘要写入 `forecast_results.summary`
8. 将完整预测序列写入 `detail_path`

要求：

- 后端负责时间范围校验
- 后端负责历史长度不足时的业务错误转换
- 后端负责把模型原始预测数组转换为公共 API 所需的 `summary` 与 `series`
- 未来预测接口不返回评估指标

## 7.5 回测流程

目标：实现 `POST /api/v1/datasets/{id}/forecasts/backtest`

流程：

1. 读取回测区间
2. 抽取历史上下文 + 回测区间实际值
3. 调用模型客户端 `Backtest`
4. 返回 `actual + predicted + metrics`
5. 第一版可不落库，直接返回结果

说明：

- 回测是实验与验证接口，不强制写入 `forecast_results`
- 回测指标统一使用 `MAE / RMSE / SMAPE / WAPE`
- 如果后续需要保存回测记录，再新增专门结果表，不建议先混入未来预测表

## 7.6 规则建议生成流程

目标：实现 `POST /api/v1/datasets/{id}/advices/generate`

规则建议由 Go 本地生成，不依赖 LLM。

输入上下文：

- 统计分析摘要
- 峰谷平占比
- 最大负荷时段
- 最新分类结果
- 最新预测摘要

输出要求：

- 3 到 8 条建议
- 每条建议必须有触发依据与可执行动作

流程：

1. 聚合分析、分类、预测摘要
2. 运行规则建议生成器
3. 将正文写入 `outputs/advices/*.json`
4. 将摘要写入 `energy_advices`

## 7.7 智能问答流程

目标：实现 `POST /api/v1/agent/ask`

流程：

1. 校验 `dataset_id`、`session_id`、`question`
2. 读取统计分析结果
3. 读取最近历史用电摘要
4. 读取最新分类结果
5. 读取指定时间范围或最新一次预测摘要
6. 读取规则建议
7. 读取默认 LLM 配置与提示词模板
8. 组装结构化上下文
9. 通过 `agentclient` 以 HTTP/JSON 调用 Python Robyn 后端中的 LangChain 智能体路由
10. 保存问答消息
11. 返回 `answer`、`citations`、`actions`

功能范围：

- 第一阶段仅支持单体节能问答智能体，不引入多 Agent 编排
- Go 主服务负责数据查询、上下文聚合、会话管理、消息持久化和降级
- Python Robyn 后端负责承载问答生成，可在其智能体模块内部使用 LangChain 函数调用完成固定工具编排
- `citations` 必须作为结构化依据返回，供前端解释展示与降级复用

第一阶段补强能力：

- 数据概览问答
- 分类结果解释
- 预测结果解读
- 规则建议重述
- 行动清单生成
- 会话记忆
- 降级问答

降级策略：

- 智能体超时、鉴权失败、网络失败时，不中断接口
- 直接返回基于规则建议的降级响应
- `degraded = true`
- `error_reason` 写入明确错误码

结论：

- 智能问答是增强功能
- 规则建议始终是兜底基础能力

## 7.8 报告导出流程

目标：实现 `POST /api/v1/datasets/{id}/reports/export`

第一版只做 Excel。

流程：

1. 聚合数据集摘要、统计分析、分类结果、建议
2. 生成 Excel 文件
3. 文件落盘到 `outputs/reports/`
4. 写入 `reports`
5. 通过下载接口返回文件流

说明：

- HTML/PDF 后续再加
- 导出失败返回 `EXPORT_FAILED`

## 8. 对外服务占位接口

Python Robyn 后端后续实现，但 Go 主服务现在就应先抽象接口，避免业务代码直接写死 HTTP 调用细节。

## 8.1 模型服务客户端接口

建议在 `internal/integration/modelclient` 中定义：

```go
type Client interface {
    Health(ctx context.Context) error
    GetModelInfo(ctx context.Context) (*ModelInfo, error)
    PredictClassification(ctx context.Context, req *PredictRequest) (*PredictResponse, error)
    Forecast(ctx context.Context, req *ForecastRequest) (*ForecastResponse, error)
    Backtest(ctx context.Context, req *BacktestRequest) (*BacktestResponse, error)
}
```

第一版提供两个实现：

- `httpClient`：真实调用 Python Robyn 后端中的模型路由
- `stubClient`：开发期返回固定数据

## 8.2 智能体服务客户端接口

建议在 `internal/integration/agentclient` 中定义：

```go
type Client interface {
    Health(ctx context.Context) error
    Ask(ctx context.Context, req *AskRequest) (*AskResponse, error)
}
```

第一版也提供两个实现：

- `httpClient`
- `stubClient`

服务边界建议：

- `httpClient` 调用同一个 Python Robyn 服务中的智能体路由
- Go 不直接嵌入 LangChain 运行时
- Python 智能体模块尽量保持无状态，不直接操作 MySQL

## 8.3 占位阶段的明确要求

- 业务层只能依赖接口，不能依赖具体 HTTP 实现
- handler 不直接调用外部服务
- 所有外部调用必须带超时控制
- 所有外部调用错误必须转换为统一业务错误码

## 10. 统一错误处理要求

后端内部必须区分三类错误：

1. 请求参数错误
2. 业务状态错误
3. 外部依赖错误

映射要求：

- 参数问题：`400` 或 `422`
- 资源不存在：`404`
- 状态冲突：`409`
- 外部服务失败：`502`
- 未知错误：`500`

必须统一返回：

- `code`
- `message`
- `data`
- `request_id`
- `timestamp`

## 12. 当前结论

后端现在不应该等待 Python Robyn 后端全部完成后再动工，而应先完成以下三件事：

1. 固定 Go 主服务内部分层与主流程
2. 先把导入、分析、规则建议、报告这些本地能力做起来
3. 用 `modelclient` 和 `agentclient` 预留接口，让分类、预测、问答后续平滑接入同一个 Robyn 服务

这条路线的优点是：

- 不阻塞前后端并行开发
- 不阻塞数据库与文件存储落地
- 不阻塞规则分析与规则建议先交付
- 后续模型和智能体接入时，只替换 integration 层，不推翻主业务流程
