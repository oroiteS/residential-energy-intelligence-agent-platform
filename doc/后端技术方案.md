## 1. 文档目标

本文件用于解决两个问题：

1. 后端主服务到底采用什么技术栈与工程方案
2. 后续后端技术文档应如何分层，避免内容重复和边界混乱

这份文档是后端开发的技术基线。后续补充 API 文档、实现代码、测试文档时，应优先与本文件保持一致。

## 3. 后端定位与原则

Go 主服务是系统的业务编排中心，不是模型训练服务，也不是 LLM 服务。

后端必须遵守以下原则：

- 单用户、本地部署、零登录
- JSON 优先，文件上传和下载除外
- 元数据入库，大对象落盘
- 对前端提供稳定 REST API
- 对 Python Robyn 后端采用接口化集成，按模型能力与智能体能力进行逻辑隔离，避免强耦合
- LLM 失败必须降级，不影响主流程
- 第一版优先稳定闭环，不引入过重基础设施

## 4. 技术栈选型

## 4.1 核心语言与运行时

- 语言：Go `1.25.7`
- 运行模式：单体后端服务
- 部署形态：本地进程运行，默认绑定 `127.0.0.1`

选择理由：

- Go 适合做 API 编排、文件处理、并发任务和外部服务调用
- 单体服务更适合当前毕业设计范围，减少部署复杂度

## 4.2 Web 框架

- 选择：`Gin`

选择理由：

- 与现有项目文档保持一致
- 路由、中间件、参数绑定、文件上传能力成熟
- 足够轻量，适合当前项目规模

不选的方案：

- 不引入更重的 Web 框架
- 不使用生成式全家桶框架，避免学习和维护成本

## 4.3 数据访问层

- 选择：`GORM`
- 数据库：`MySQL 8.0`
- 驱动：`gorm.io/driver/mysql`

选择理由：

- 当前项目实体较多，使用 GORM 更适合快速建立模型、查询与关联关系
- 对当前毕业设计阶段而言，能减少样板代码，更利于推进整体后端开发速度
- 后续分页、过滤、状态更新、基础 CRUD 可以更快形成闭环

执行原则：

- GORM 用于主数据访问层
- 复杂查询允许使用 `Raw SQL`
- 不强依赖自动迁移，表结构仍以 `schema.sql` 为准
- 模型定义必须严格对齐现有数据库文档和建表脚本

## 4.4 配置管理

- 环境变量：`os.Getenv`
- `.env` 加载：`github.com/joho/godotenv`
- 系统级动态配置：数据库表 `system_config`

职责划分：

- `.env` 负责运行环境与基础连接配置
- `system_config` 负责峰谷配置、模型历史窗口、提示词模板等业务配置

## 4.5 日志与可观测性

- 日志：`go.uber.org/zap`
- Gin 请求日志：`github.com/gin-contrib/zap`
- GORM 日志：自定义适配到 `zap`
- 请求追踪：自定义 `request_id`
- 错误分类：统一业务错误码

选择理由：

- `zap` 功能成熟、性能高，结构化日志能力强
- 与 Gin 集成完善，适合统一请求日志、错误日志和外部调用日志
- 便于后续统一接入 SQL 日志与慢查询日志

第一版日志策略：

- 开发环境默认 `info`
- 慢 SQL 单独输出 `warn`
- 外部服务错误输出 `error`
- `RecordNotFound` 不默认记为错误日志

第一版要求：

- 每个请求都输出 `request_id`
- 外部服务调用失败必须带服务名、目标地址、超时信息
- 数据导入、分析重算、导出报告必须记录关键状态日志

## 4.6 HTTP 客户端

- 选择：标准库 `net/http`
- 封装位置：`internal/integration/modelclient`、`internal/integration/agentclient`

要求：

- 每个外部调用必须带超时
- 禁止在 handler 中直接发 HTTP 请求
- 必须统一转换外部错误为业务错误码

## 4.7 数值处理

- 十进制精度：`github.com/shopspring/decimal`

适用场景：

- 电量、比例、功率汇总值
- Excel 导出中的数值格式化

## 4.8 Excel 导出

- 选择：`github.com/xuri/excelize/v2`

选择理由：

- 与项目文档一致
- 足够支持统计表、日聚合表、曲线数据表导出

## 4.9 参数校验

- JSON/表单绑定：Gin 自带绑定能力
- 结构体字段校验：`github.com/go-playground/validator/v10`

要求：

- 所有请求 DTO 与查询参数统一做校验
- 业务校验不混入 handler，应在 service 层补充处理

## 4.10 测试栈

- 单元测试：标准库 `testing`
- HTTP 测试：`net/http/httptest`
- 断言辅助：可选 `github.com/stretchr/testify`

原则：

- 测试执行时间单次不超过 `60s`
- 优先做可重复的单元测试和小样本集成验证

## 4.11 异步任务模型

- 第一版选择：进程内任务执行器
- 不引入消息队列
- 不引入分布式任务系统
- `Asynq` 作为后续扩展方案，不作为当前必选栈

适用任务：

- 数据集导入
- 统计分析重算
- 报告导出

原因：

- 当前是本地单用户系统，任务规模和并发要求有限
- 使用进程内任务模型可快速形成闭环
- 不额外引入 Redis，能降低部署和调试复杂度

后续升级条件：

- 如果后面明确需要任务重试、优先级、定时任务、可视化任务管理，再考虑引入 `Asynq`

## 5. 分层架构设计

建议后端采用“接口层 -> 服务层 -> 仓储层 -> 基础设施层”的清晰分层。

```text
前端
  ↓
Gin Handler / Router / Middleware
  ↓
Service
  ↓
Repository / Analysis / Advice / Report
  ↓
MySQL / 文件系统 / Python Robyn 后端
```

### 5.1 接口层

职责：

- 路由注册
- 请求参数解析
- DTO 校验
- 响应结构封装
- 错误码映射

禁止：

- 直接写 SQL
- 直接做复杂业务编排
- 直接调用外部模型或智能体 HTTP 接口

### 5.2 服务层

职责：

- 执行业务主流程
- 组织事务
- 调用 repository、analysis、advice、report、integration
- 执行状态流转与业务校验

这是后端的核心层。

### 5.3 仓储层

职责：

- 封装数据库查询
- 封装分页、列表、详情、创建、更新、删除
- 隔离 SQL 细节

要求：

- 一个 repository 只处理一类聚合或主实体
- 不把复杂业务逻辑写进 repository

### 5.4 领域能力层

建议拆出以下纯业务能力模块：

- `analysis`
  - 统计分析
  - 峰谷平分析
  - 质量摘要生成
- `advice`
  - 规则建议生成
  - 问答降级建议
- `report`
  - Excel 文件生成

### 5.5 集成层

建议拆出以下外部客户端：

- `modelclient`
- `agentclient`

要求：

- 统一封装请求结构与响应结构
- 提供 `http` 与 `stub` 两种实现
- 提供超时、错误转换、日志记录
- 两个客户端允许指向同一个 Robyn 服务的不同路由前缀

## 6. 推荐目录结构

```text
backend/
├── cmd/api/
│   └── main.go
├── internal/
│   ├── bootstrap/
│   ├── config/
│   ├── http/
│   │   ├── dto/
│   │   ├── handler/
│   │   ├── middleware/
│   │   └── router/
│   ├── service/
│   ├── repository/
│   ├── domain/
│   ├── analysis/
│   ├── advice/
│   ├── report/
│   ├── integration/
│   │   ├── modelclient/
│   │   └── agentclient/
│   ├── storage/
│   └── job/
└── pkg/
    ├── apperror/
    ├── response/
```

说明：

- `domain` 存放核心实体和枚举，不直接等同于数据库模型

- `storage` 负责文件系统读写和路径管理
- `job` 负责进程内异步任务

- 数据集元信息
- 分析摘要
- 分类摘要
- 预测摘要（写入 `forecast_results.summary`）
- 会话与消息摘要

文件系统保存：

- 原始上传文件

- 质量报告 JSON
- 分析详情 JSON
- 预测明细 JSON
- 建议正文 JSON
- 长消息内容
- Excel 报告

补充约定：

- Python Robyn 后端的预测模块对预测任务返回原始 `predictions` 数组
- Go 主服务负责生成 `ForecastSummary`，并缓存到 `forecast_results.summary`
- 预测详情文件保存带时间戳的完整序列，供详情页与报告导出使用

## 8. API 文档开发前置约束

后续开发后端 API 文档时，必须遵守以下技术约束：

### 8.1 契约稳定优先

- 字段名统一使用 `snake_case`
- 统一响应结构不随接口变化
- 列表接口统一分页结构
- 文件下载接口不包裹 JSON

### 8.2 状态机优先

接口文档必须体现状态流转，而不是只写 URL。

至少要明确：

- `datasets.status`
- 智能问答的降级响应

### 8.3 先写摘要字段，再写详情文件

对分析、预测、建议、报告等接口：

- API 返回高频摘要字段
- 详细大对象通过 `*_path` 读取或下载

### 8.4 占位接口提前设计

即使 Python Robyn 后端暂未实现，也应在 API 文档中保留：

- 请求体结构
- 正常响应结构
- 外部服务失败时的错误响应结构
- 降级响应结构

## 9. 当前技术决策

当前阶段冻结以下决策：

### 9.1 采用的技术

- Go `1.25.7`
- Gin
- MySQL 8.0
- GORM
- `gorm.io/driver/mysql`
- `godotenv`
- Zap
- `gin-contrib/zap`
- `shopspring/decimal`
- `excelize`
- `validator/v10`

### 9.2 明确不采用的方案

- 不采用消息队列
- 不采用微服务拆分
- 不在第一版引入 Redis
- 不在第一版引入容器编排
- 不在第一版引入复杂权限认证

### 9.3 模型与智能体接入策略

- 主业务代码只依赖接口
- 先用 stub 客户端推动联调
- 后续替换为真实 HTTP 客户端

## 11. 当前结论

在开始继续细化后端 API 文档之前，技术基线应当先固定为：

- 用 **Gin + GORM + MySQL** 作为 Go 主服务基础栈
- 用 **Zap + 统一错误码 + 进程内异步任务 + 文件落盘** 作为工程方案
- 用 **stub 模型客户端 / stub 智能体客户端** 作为当前联调方案
- 用 **技术方案文档 + 开发规划文档 + API 契约文档 + 数据库文档** 组成后端文档体系

这样做的目的，是让后续 API 文档开发和实际代码实现都基于同一套技术口径推进，避免边写接口文档边改技术基线。
