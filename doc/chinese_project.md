# 居民用电分析与节能建议系统

---

## 一、项目概述

### 1.1 项目定位

本项目是一个面向居民的用电数据分析与节能建议系统，核心目标是：

- 满足毕业设计：完成数据导入、预处理、统计分析、行为分类、可视化仪表盘、报告导出与节能建议
- 展示技术深度：采用规则标签定义 4 类居民用电模式，使用 TCN 完成日级行为分类，并对比 LSTM 与 Transformer 两类模型完成负荷预测
- 控制复杂度：采用本地自部署、零登录；移除异常告警、模拟数据生成等高维护模块

### 1.2 核心价值

| 维度  | 价值体现                                    |
| --- | --------------------------------------- |
| 实用性 | 让用户理解用电结构，得到可执行的节能建议                    |
| 技术性 | 完整 ML 工程链路：15 分钟预处理→规则标签生成→TCN 分类→LSTM/Transformer 预测→推理服务→前端展示 |
| 工程性 | Go 后端 + MySQL + Python 服务分层协作，兼顾工程实现与模型落地 |
| 可复现 | 明确数据版本、模型版本、实验对比与评估指标                   |

### 1.3 核心性能指标（验收向）

| 指标   | 要求                                         |
| ---- | ------------------------------------------ |
| 分类指标 | 行为分类准确率 ≥ 85%（或在固定测试集上给出 accuracy/F1 并可复现） |
| 预测指标 | 在固定回测集上给出 MAE/RMSE/SMAPE/WAPE，并可复现实验结果       |
| 系统响应 | 统计分析 API ≤ 3 秒（常规数据量）                      |
| 推理速度 | 单次推理（单窗口/单日）≤ 2 秒                          |
| 运行环境 | 本地 RTX 3050 Laptop 可训练与推理；CPU 可运行基础分析      |

### 1.4 部署模式

| 项目   | 说明                                   |
| ---- | ------------------------------------ |
| 部署方式 | 单用户个人部署（Self-hosted）                 |
| 认证方式 | 无需登录认证                               |
| 数据存储 | MySQL + 文件系统（数据集文件/模型权重/导出报告）        |
| 网络绑定 | 默认 127.0.0.1（本地访问）                   |
| 隐私保护 | 数据完全本地化；智能体依赖外部 LLM API，但失败时必须自动降级为规则建议 |

## 四、功能需求（分层设计）

### 4.1 模块1：数据导入与预处理（必做）

功能：

- 上传 CSV/Excel 格式用电数据
- 字段要求：最少包含时间戳 + 总用电量（kWh 或 W/Wh；导入时明确单位）
- 可选包含任意数量的电器子表功率列；项目不要求电器列固定为 9 个
- 自动清洗：
  - 缺失值处理（插值/剔除策略可配置）
  - 异常值处理（IQR/阈值裁剪/标记）
  - 时间对齐到 15 分钟粒度（默认 15min）
  - 统一时区与时间格式解析
  - 从电器子表衍生辅助特征：`active_appliance_count`、`burst_event_count`

输出：

- 数据集元信息：起止时间、总条数、缺失率、采样间隔、单位
- 清洗后 15 分钟序列：用于统计、规则分类与建模

### 4.2 模块2：基础统计分析（必做）

至少包含：

- 总用电量、日均用电量
- 峰/谷/平时段占比（规则可配置）
- 最高/最低负荷及发生时段
- 趋势分析：按天/周聚合
- 典型日曲线：24小时平均负荷

### 4.3 模块3：行为模式识别（必做 + 亮点）

#### 4.3.1 行为类别（规则冻结）

最终冻结为以下 4 个稳定标签，前后端、Python Robyn 后端、数据库展示统一使用：

- `day_high_night_low`：白天高晚上低型
- `day_low_night_high`：白天低晚上高型
- `all_day_high`：全天高负载型
- `all_day_low`：全天低负载型

说明：

- `predicted_label` 一律返回上述英文稳定标识
- 前端负责映射为中文展示文案
- 如无强制需求，不再新增或重命名标签，避免影响训练数据、接口契约和前端渲染逻辑

#### 4.3.2 规则标签生成（必须）

目的：

- 直接按照业务定义生成稳定、可解释、可复现的监督标签
- 为后续 TCN 分类训练提供统一标签来源

实现方案：

- 以“单天”为样本单位
- 白天时间段固定为 `08:00-18:00`
- 夜间时间段固定为 `18:00-08:00`
- 计算：
  - `day_mean`
  - `night_mean`
  - `full_mean`
- 依据全训练集日样本分位数和时段均值比值生成标签：
  - `high_threshold = full_mean` 的 `P75`
  - `low_threshold = full_mean` 的 `P25`
  - `ratio_threshold = 1.2`
- 主规则：
  - 白天、夜间均高于 `high_threshold` -> `all_day_high`
  - 白天、夜间均低于 `low_threshold` -> `all_day_low`
  - `day_mean / night_mean >= ratio_threshold` -> `day_high_night_low`
  - `night_mean / day_mean >= ratio_threshold` -> `day_low_night_high`
- 边界样本采用得分兜底，保证每个样本都能落入 4 类之一

#### 4.3.3 深度学习分类（必须）

模型：TCN 分类器

- 输入：单天样本，形状为 `96 x 3`
- 三个通道固定为：`Aggregate`、`active_appliance_count`、`burst_event_count`
- 输出：类别概率（softmax）+ 预测标签

训练策略：

- 标注来源：统一使用规则标签生成器输出的监督标签
- 数据增强：时间平移、噪声注入、滑动窗口
- 评估指标：accuracy、macro-F1（建议同时给混淆矩阵）
- 可复现：固定随机种子、记录数据划分与模型超参

推理集成：

- Python Robyn 后端提供分类推理接口
- Go 服务调用并缓存结果（可选）用于前端展示

#### 4.3.4 时序预测（必须）

模型 A：LSTM 预测器

- 输入：最近 3 天的 15 分钟多变量序列，形状为 `288 x 3`
- 输出：下一天的 `96` 个 `Aggregate` 预测点

模型 B：Transformer 预测器

- 采用面向时序预测的 Transformer 结构，不使用推荐式离散序列建模
- 与 LSTM 预测器共享相同输入窗口、目标区间与回测设置，保证实验对比公平

评估与接口要求：

- 预测请求统一由 `forecast_start / forecast_end / granularity` 驱动
- 回测接口单独输出 `actual + predicted` 与 `MAE / RMSE / SMAPE / WAPE`
- 未来预测接口不返回评估指标，避免与回测语义混淆
- Python Robyn 后端当前返回原始预测序列；Go 主服务负责补齐时间戳、生成稳定摘要结构并落库
- 未来预测结果需额外整理为稳定摘要结构，供前端展示与大模型建议共用
- 预测训练默认按 `by_house` 切分；`random` 仅用于开发阶段快速验证，不作为最终泛化结论

预测摘要结构（冻结）：

- `forecast_start` / `forecast_end` / `granularity`
- `predicted_avg_load_w`：预测区间平均负荷
- `predicted_peak_load_w`：预测区间峰值负荷
- `forecast_peak_periods`：预测高负荷时段列表
- `predicted_peak_ratio` / `predicted_valley_ratio` / `predicted_flat_ratio`
- `risk_flags`：风险提示标记，例如 `evening_peak_risk`、`night_load_risk`

说明：

- `ForecastSummary` 由 Go 主服务根据模型原始预测序列生成
- 当前 LSTM 原始输出固定为长度 `96` 的 `predictions` 数组
- 后续新增预测模型只要保持该原始输出契约一致，即可复用同一后端摘要生成逻辑

### 4.4 模块4：可视化仪表盘（必做）

页面最小集：

- 数据集管理：上传、列表、查看元信息
- 数据集详情：
  - 核心统计卡片
  - 负荷曲线（按天/周/典型日切换）
  - 峰谷占比图
  - 行为分类结果（类别 + 置信度）
  - 节能建议面板
    - 节能问答对话框（LangChain 智能体，异常时展示降级提示）
- 报告导出入口（Excel/可选HTML）

### 4.5 模块5：节能建议生成（必做，规则模板）

基于：

- 统计分析结果、峰时用电占比、夜间占比、最高负荷时段、行为分类结果、时序预测结果
- 智能体建议链路中，额外提供“过去几天真实用电摘要 + 未来时间范围预测摘要 + 居民行为分类结果”作为联合上下文

输出形式：

- 3-8条可执行建议
- 每条建议包含：触发依据（可解释）+ 操作建议

### 4.6 模块6：报告导出（必做：Excel；可选：HTML/PDF）

必做：

- 导出 Excel：统计指标表、峰谷表、日聚合表、曲线数据表

可选：

- 导出 HTML 报告（包含图表截图或前端打印）
- PDF 作为增强，不作为 v6.2 必交付项

## 五、数据输入规范（验收关键）

### 5.1 支持文件

- CSV
- Excel(xlsx)

### 5.2 必需字段与别名映射

文件必须包含：时间戳 + 用电量（允许自动识别常见列名）。

- 时间戳列：`timestamp` / `time` / `datetime` / `日期时间`
- 用电量列：`kwh` / `usage` / `value` / `电量`

若未命中：前端提供手动选择列映射（强烈建议实现，显著降低导入失败率）。

电器子表列为可选输入：

- 允许任意数量列，不限制为 `Appliance1~Appliance9`
- 可按列名前缀、列映射或用户手动选择识别
- 识别出的电器列统一参与 `active_appliance_count` 与 `burst_event_count` 构造
- 若未提供电器列，系统仍可完成导入与基础统计，但分类/预测能力受限

### 5.3 时间格式支持

- ISO8601：`2026-02-06T13:00:00`
- 常用格式：`YYYY-MM-DD HH:MM:SS`、`YYYY/MM/DD HH:MM`
- Excel 序列日期：自动识别

### 5.4 单位约定

- 系统内部统一存储为 kWh
- 若输入为 Wh：除以 1000
- 若输入为 W：
  - 在 15 分钟粒度下：`kWh = W * 0.25 / 1000`
  - 若原始数据不是 15 分钟粒度，必须先重采样再完成单位换算

## 六、数据预处理与质量检查

### 6.1 15 分钟粒度对齐

- 时间对齐：`floor_to_15min`
- 同一 15 分钟内多条记录聚合方式（默认 `mean`）：
  - `mean`：适用于 REFIT 一类瞬时功率采样
  - `sum`：仅在原始输入为能量累计值时启用
- 15 分钟缺口处理（可配置，默认线性插值）：
  - 线性插值
  - 置零
  - 严格模式：删除缺失时间窗所在日期或拒绝导入

### 6.2 辅助特征构造（建模必做）

- `active_appliance_count`
  - 定义：同一时刻已识别电器列中功率大于 `10W` 的电器数量
  - 目的：刻画该时刻活跃电器数量与家庭活动强度
- `burst_event_count`
  - 定义：已识别电器列中，相邻采样点功率增量大于 `300W`，且高功率持续不超过 `2` 个 15 分钟窗口的突发启动事件数量
  - 目的：刻画电热水壶、微波炉等短时高功率设备带来的瞬时行为信号

### 6.3 异常值处理（不做告警，但做清洗/标记）

- 默认策略：IQR 或阈值裁剪（winsorize），并记录被裁剪点数
- 输出异常统计：最小/最大、P95/P99、异常点数量

### 6.4 数据质量摘要（导入后必须生成并展示）

- 时间范围（start/end）
- 总记录数、应有 15 分钟点数、缺失率
- 推断采样间隔（建模默认应为 15min）
- 重复时间戳数量
- 清洗策略记录与清洗后有效点数

## 八、负荷模式识别（满足任务书 + 升级版）

### 8.1 任务书最低要求（必须满足）

通过规则标签直接判断用户属于以下 4 类之一：

- 白天高晚上低型
- 白天低晚上高型
- 全天高负载型
- 全天低负载型

实现：

- 以“单天”为样本单位，基于白天均值、夜间均值、全天均值直接生成标签
- 规则阈值由日样本 `full_mean` 的分位数统计得到
- 规则标签直接作为基础标签层，供分类模型训练与对比实验复用

### 8.2 升级版分类（以实现为准）

系统分类固定为 4 类：

- 白天高晚上低型
- 白天低晚上高型
- 全天高负载型
- 全天低负载型

输出要求：

- label：类别
- confidence：置信度/概率（softmax 或规则得分归一化）
- explain（建议）：一句话解释（例如“白天均值 0.31 高于夜间均值 0.18，且比值超过 1.2”）

### 8.3 深度学习对比（升级亮点，建议做）

- TCN 分类器：单天 `96 x 3` 时序输入 -> 分类输出

评估要求：

- 固定数据划分与随机种子
- accuracy、macro-F1、混淆矩阵
- 记录推理耗时与模型参数量（可选）

### 8.4 时序预测对比（升级亮点，建议做）

- LSTM 预测器：基于最近 3 天的历史窗口输出下一天 96 点预测
- Transformer 预测器：基于最近 3 天的多变量序列输出下一天 96 点预测

评估要求：

- 默认按家庭切分训练/验证/测试集，以评估新家庭泛化能力
- 可额外保留 `random` 冒烟实验，但不得替代跨家庭评估
- 使用独立回测区间进行对比
- 指标统一为 MAE、RMSE、SMAPE、WAPE
- 未来预测与回测接口语义分离，避免把评估指标混入真实未来预测结果

## 九、建议与智能问答模块

### 9.1 节能建议生成（必做，规则模板，可解释）

建议生成依据：

- 峰时占比、谷时占比
- 最大负荷出现时段
- 行为分类标签（如夜间型/高耗能型）
- 时序预测结果（未来高负荷时段、未来峰谷迁移趋势）
  -（可选）波动性指标、待机占比估计

输出规则：

- 生成 3-8 条建议
- 每条建议包含：
  - 触发依据（指标值 + 阈值/对比）
  - 可执行建议（明确动作）

示例模板：

- 条件：`peak_ratio > 0.45`
  - 依据：峰时用电占比 52%，高于阈值 45%
  - 建议：将洗衣、充电、热水器加热等调整到谷时段执行，以降低费用并平滑负荷。

### 9.2 节能问答智能体（必做，LangChain + OpenAI 格式 API）

定位：

- 在规则建议基础上，提供“解释/问答/行动清单整理”能力
- 不参与硬指标计算（统计、峰谷、分类结果均由系统产生）
- 采用 Python Robyn 后端中的 LangChain 单体问答模块，不做复杂多 Agent 编排
- 属于正式交付模块，前端需提供入口，后端需提供稳定接口
- LLM 不可用时必须自动降级，不影响系统核心功能验收

系统边界：

- Go 主服务负责读取数据库、加载规则建议、组装结构化上下文、管理会话与消息、执行降级
- Python Robyn 后端负责承载 `classification`、`forecast`、`agent` 三类能力，其中智能体模块基于上下文和用户问题生成回答，不直接访问数据库
- 服务间通信统一使用 HTTP/JSON
- 智能体内部允许使用 LangChain 函数调用，但函数范围仅限项目内固定工具，不扩展为通用联网助手

接口规范（写死）：

- 仅支持 OpenAI API 格式（OpenAI-compatible）对话接口
- 不适配 Claude/Gemini/Ollama 原生协议；如需使用其他模型，要求通过 OpenAI-compatible 网关提供同格式 API

配置（不写死在代码里）：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`
- `LLM_TEMPERATURE`（默认 0.2）
- `LLM_TIMEOUT_SECONDS`（默认 60）

智能体输入上下文（防幻觉关键）：

- 系统向智能体提供结构化 JSON 上下文，仅包含本数据集汇总指标与结论，例如：
  - 时间范围、缺失率、清洗策略
  - 总用电、日均、最大/最小 15 分钟负荷及时间
  - 峰谷配置与峰/谷/平用电占比
  - 过去几天真实用电情况摘要（例如最近 3-7 天负荷变化、异常高值时段、峰谷迁移）
  - 分类结果 label/confidence/explain
  - 未来预测结果摘要（例如未来高负荷时段、未来峰谷迁移趋势、未来总量估计）
  - 规则建议列表（含触发依据）

第一阶段补强功能（冻结）：

- 数据概览问答：回答总用电、日均、峰谷占比、最大负荷时段、趋势概览
- 分类结果解释：解释 `predicted_label`、`confidence` 与昼夜均值差异
- 预测结果解读：解释未来高负荷时段、风险标签与负荷峰值
- 规则建议重述：把规则建议转成更贴合用户提问的自然语言回答
- 行动清单生成：按“今天就能做 / 本周可调整 / 长期优化”整理建议动作
- 引用依据输出：每次回答返回 `citations`
- 会话记忆：支持同一 `session` 下连续追问
- 降级问答：LLM 失败时基于规则建议生成可读回答

第一阶段函数调用范围（冻结）：

- 查询数据集摘要
- 查询最新分类结果
- 查询最新预测摘要
- 查询规则建议
- 读取会话历史
- 读取系统配置

提示词组装约定（建议写死到实现中）：

- 服务端负责组装结构化上下文，不直接把原始数据库记录拼接给大模型
- 提示词语义应明确包含三类信息：
  - 过去几天的实际用电情况
  - 未来时间范围内的预测用电情况
  - 居民当前的用电行为分类
- 建议提示词模板可采用如下语义：
  - “这是某居民过去几天的实际用电情况、未来一段时间的预测用电情况，以及该居民的用电行为分类。请基于这些信息给出具体、可执行、可解释的节能建议，并指出建议所依据的关键指标或时段。”

输出约束：

- 回答必须引用上下文指标；缺少数据时明确说明“无法判断”
- 输出建议与解释应可追溯到规则建议或指标依据，避免凭空编造

API 设计（建议）：

- Go 对前端：`POST /api/v1/agent/ask`
- 请求：`dataset_id`、`question`、`history`（可选）
- 响应：`answer`、`citations`（结构化回答依据）、`actions`（可选建议清单）

`citations` 字段定义：

- 用于表达当前回答所依据的指标、分类结果、预测结论或规则建议
- 不是外部链接或论文引用，而是面向前端展示的结构化证据列表
- 每项至少包含：`key`、`label`、`value`

降级策略（必须）：

- LLM 超时/失败：前端仅展示规则建议，提示“智能问答暂不可用”，不影响分析、分类、导出流程
- API 返回异常、鉴权失败、限流、网络中断时：统一记录错误日志并返回可解释错误状态，不得导致主流程崩溃
- 智能体回答失败后：允许用户重试，但默认回退到规则建议与已有分析结论

## 十一、API 与服务边界（实现约束）

### 11.1 Go（Gin）对前端 API（建议）

- 数据集：
  - 上传、列表、详情、删除
- 分析：
  - 基础统计、峰谷分析、图表数据
- 分类：
  - 触发推理、获取最新结果
- 预测：
  - 触发未来预测、执行回测、获取预测详情
- 导出：
  - 导出 Excel（返回下载链接或直接下载）
- 智能体：
  - `POST /api/v1/agent/ask`

### 11.2 Python（Robyn）后端 API（建议）

- `GET /internal/model/v1/health`
- `GET /internal/model/v1/model/info`：模型版本、类别定义、输入规格（窗口长度、单位等）
- `POST /internal/model/v1/predict`：输入序列/特征 -> 输出分类 label 与原始概率结果
- `POST /internal/model/v1/forecast`：输入历史序列 + 预测时间范围 -> 输出未来原始预测序列
- `POST /internal/model/v1/backtest`：输入带实际值的回测区间 -> 输出预测序列与评估指标
- `GET /internal/agent/v1/health`
- `POST /internal/agent/v1/ask`：输入结构化上下文与用户问题 -> 输出 `answer/citations/actions`

说明：

- Go ↔ Python：HTTP + JSON
- Python 侧统一为一个 Robyn 服务，对外暴露 `model` 与 `agent` 两组内部路由
- 输入输出规范通过 `/internal/model/v1/model/info` 固化，避免前后端漂移

## 十四、验收清单（功能验收口径）

- 能上传 CSV/Excel 并正确解析时间戳与用电量（15 分钟粒度）
- 导入后展示数据质量摘要（缺失/重复/范围/间隔/清洗策略）
- 自动生成基础统计与峰谷平分析（并展示峰谷配置）
- 仪表盘展示至少两张核心图：负荷曲线、峰谷比例饼图（建议包含趋势图/典型日图）
- 能给出负荷模式识别结果（至少满足白天/夜间/均衡；升级版可输出 5 类）
- 能按时间范围输出未来预测结果，并提供独立回测指标（MAE/RMSE/SMAPE/WAPE）
- 能输出 3-8 条可解释节能建议（含触发依据）
- 必须提供节能问答入口，且在 LLM API 超时、报错或不可用时能自动降级并继续使用核心功能
- 支持导出 Excel 报告文件
- LLM 不可用时系统不崩溃且可正常导出报告（降级可用）
